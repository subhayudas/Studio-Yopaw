D. Task B — Extra Attendees UI

The Type Definition

We defined a local type inside PricingSection to represent each extra person:

TypeScript
type ExtraAttendee = { name: string; waiverAccepted: boolean }
const [extraAttendees, setExtraAttendees] = useState<ExtraAttendee[]>([])
🔑
Why bundle name and waiverAccepted into one object per attendee? The alternative would be two parallel arrays — string[] for names and boolean[] for waivers. Parallel arrays require careful index synchronisation and are a common source of off-by-one bugs. A single array of objects keeps each attendee's data co-located — you can't accidentally mismatch them.

The Extra Attendees UI Block

This renders below the phone input in the yin (Regular Class) contact form. The key logic: add up to 4, remove by index, update name or waiver by index:

E. Concept: Array State in React

React's useState works with any JavaScript value — including arrays of objects. But arrays have a subtle requirement: you must always replace the array with a new array. React uses reference equality to detect changes. If you mutate the existing array in place, React sees the same reference and skips the re-render.

Immutable Array Update Patterns

TypeScript — the four patterns we used
// ❌ WRONG — mutates in place, React won't re-render
extraAttendees[0].name = 'Alice'
setExtraAttendees(extraAttendees)  // same reference!

// ✅ ADD — spread existing + append new item
setExtraAttendees(prev => [...prev, { name: '', waiverAccepted: false }])

// ✅ REMOVE — filter keeps everything except index i
setExtraAttendees(prev => prev.filter((_, j) => j !== i))

// ✅ UPDATE — spread into new array, replace item at index i
setExtraAttendees(prev => {
  const next = [...prev]          // new array, same item references
  next[i] = { ...next[i], name }  // new object at index i
  return next
})

// ✅ CLEAR — just return an empty array
setExtraAttendees([])
📖
Real-world context. This "immutable update" pattern is the same reason Redux Toolkit exists — it lets you write apparent mutations but secretly produces new references via Immer. In this codebase we're too small for Redux, so we use the spread pattern directly. In production apps like Notion or Linear, the same principle drives their collaborative state machines — every change produces a new state snapshot that can be diffed, compared, or time-traveled.

G. Concept: Computed disabled Props

The disabled attribute on a button accepts any boolean expression. React evaluates it every render — as state changes, the button automatically enables or disables with zero additional code. This is the "declarative" mindset: you describe what should be true, not what to do.

TypeScript — how the validation evaluates
// Initial state (no extras added) — primary waiver controls the button
!false || [].some(...) || [].some(...)
→ true  // button DISABLED — must check primary waiver first

// Primary waiver checked, one extra added but no name yet
!true || [{ name: '', waiverAccepted: false }].some(a => !a.waiverAccepted)
→ false || true
→ true  // still DISABLED

// All waivers checked, all names filled
!true || [{ name: 'Alice', waiverAccepted: true }].some(a => !a.waiverAccepted)
→ false || false
→ false  // button ENABLED ✅

I. Task E — Per-Attendee Square Customers + Fallback

The Problem

The original code created one Square customer record (the primary booker). If 4 people attend, 3 of them have no record in Square's CRM. The diagram required one customer per attendee.

The Helper Function

TypeScript — findOrCreateCustomer
async function findOrCreateCustomer(
  gName: string, fName: string,
  emailAddr: string, phoneNum: string,
): Promise<string> {
  // Search first — don't create duplicates
  const search = await square.customers.search({
    query: { filter: { emailAddress: { exact: emailAddr } } },
  })
  const existing = search.customers?.[0]?.id
  if (existing) return existing

  // Not found — create
  const { customer } = await square.customers.create({
    idempotencyKey: randomUUID(),
    givenName: gName, familyName: fName,
    emailAddress: emailAddr, phoneNumber: phoneNum,
  })
  return customer!.id!
}

Parallel Creation + Fallback

TypeScript — Promise.all with catch fallback
// Build the full attendee list (primary + extras)
const allAttendees = [
  { givenName, familyName },
  ...(extraAttendees ?? []).map(a => {
    const parts = a.name.trim().split(' ')
    return {
      givenName:  parts[0] ?? a.name,
      familyName: parts.slice(1).join(' ') || parts[0] ?? a.name,
    }
  }),
]

let customerId: string
try {
  // Try to create all customers in parallel
  const ids = await Promise.all(
    allAttendees.map(a => findOrCreateCustomer(a.givenName, a.familyName, email, phone)),
  )
  customerId = ids[0]  // primary ID used for booking + payment
} catch (batchErr) {
  // Rate-limit or API error — fall back gracefully
  console.warn('Customer batch creation failed — falling back to primary only', batchErr)
  customerId = await findOrCreateCustomer(givenName, familyName, email, phone)
}

J. Concept: Promise.all + Graceful Fallback

TypeScript — Promise.all explained
// Sequential — slow (waits for each before starting next):
for (const a of attendees) {
  await findOrCreateCustomer(a.givenName, ...)
}

// Parallel — fast (all start simultaneously, resolve when ALL done):
const ids = await Promise.all(
  attendees.map(a => findOrCreateCustomer(a.givenName, ...))
)
// ids[0] = primary, ids[1] = extra1, ids[2] = extra2 ...

// ⚠️ Promise.all rejects the WHOLE thing if ANY one fails
// That's why we wrap it in try/catch and fall back to primary-only
📖
Real-world context. This pattern — "try to do everything in parallel, degrade gracefully on failure" — appears everywhere in distributed systems. Stripe's API does exactly this: if a batch creation call fails mid-way, they complete as many as they can and report which succeeded. The key insight is that a partial success is better than a complete failure for the user. The studio still gets its booking; the missing customer records can be fixed manually.

L. Concept: Square Order Quantity vs Custom Price

This is one of the more subtle patterns in this codebase. When you create a Square Order, you have two ways to set the price for a line item:

MethodHow it WorksWhen to Use
Catalog Reference
catalogObjectId + quantity		 Square looks up the item's price from its own Catalog. You control quantity; Square computes the total. ✅ For items you manage in Square Dashboard (services, products)
Custom Price
basePriceMoney + quantity		 You specify an arbitrary price per unit. Square doesn't reference its Catalog. ✅ For ad-hoc items with no catalog entry (mat rental, tips)
🔑
Studio Yopaw uses both patterns in the same order. The yoga class uses catalogObjectId (so the price in Square Dashboard is the authoritative price), and the mat rental uses basePriceMoney: 500n (a flat $5 ad-hoc charge with no catalog entry). Taxes are only applied to the catalog item — the mat rental line has no appliedTaxes (the studio treats it as a flat service fee).

P. Bug Stories

🐛
Bug 1: Curly Apostrophe in a TypeScript String Literal

Symptom: TypeScript compilation failed with a syntax error pointing at the FR string for pricingExtraAttendeesHelper.

Root cause: The FR string "Ajoutez jusqu'à 4 participants" was written with a curly (typographic) apostrophe ' (Unicode U+2019) instead of a straight quote '. This is a common artefact of text editors and copy-paste from word processors.

Fix: Replace with the ASCII straight apostrophe: '.

Lesson: TypeScript's string delimiters are ASCII quotes only. A curly apostrophe inside a single-quoted string literal is valid Unicode but looks identical on screen — the only clue is the cryptic parse error. Always check the raw character when debugging string syntax errors.
🐛
Bug 2: Playwright Selector Matches Two Elements

Symptom: All E2E tests crashed with "strict mode violation — locator resolved to 2 elements" when trying to click the Regular Class booking button.

Root cause: The original code used getByText('Regular Class'). But "Regular Class" appears twice in the DOM: once as the heading on the class card, and once as the text inside the booking button inside that card. Playwright's strict mode refuses to click when a selector matches multiple elements.

Fix: Target by component class instead of text:

TypeScript
// ❌ Before (matches heading + button = 2 elements)
page.getByText('Regular Class').click()

// ✅ After (matches only the card that contains the text)
page.locator('.pricing-choice-card', { hasText: 'Regular Class' }).click()
Lesson: Use semantic CSS classes or data-testid attributes to anchor Playwright selectors, not visible text alone. Text can appear multiple times. A .pricing-choice-card appears only where intended.
🐛
Bug 3: Empty Breed Mock Hides All Dates

Symptom: E2E tests timing out because the date picker showed no rows — the calendar was blank.

Root cause: The initial Playwright mock for /api/breeds returned an empty object {}. The frontend filter (effectiveDates) requires a breed entry to exist for a date before it shows that date as a session row. With no breed entries, no rows rendered.

Fix: Generate breed entries dynamically from the same slot list used for availability:

TypeScript
// ❌ Before — empty, nothing renders
route.fulfill({ json: { schedule: {} } })

// ✅ After — one breed entry per unique date from slots
for (const slot of slots) {
  const date = slot.startAt.split('T')[0]
  if (!schedule[date]) {
    schedule[date] = [{ breed: { en: 'Golden Retriever', fr: 'Golden Retriever' },
                        serviceIds: [] }]
  }
}
Lesson: When mocking related APIs, ensure their data is consistent. Availability says "these dates exist." Breeds must confirm the same dates or the UI's join logic produces nothing. Test data should be co-generated.
🐛
Bug 4: Overstrict Test Assertion

Symptom: A test for "service line item uses totalPeople as quantity" was failing even after the fix was in place.

Root cause: The assertion checked that quantity: '1' did NOT appear in booking.ts. But the mat rental line item legitimately uses quantity: '1' (mat rental is always 1 unit). Checking the whole file was too broad.

Fix: Assert the specific positive case instead of the negative:

JavaScript
// ❌ Too broad — fails because mat rental also has quantity '1'
assert.ok(!booking.includes("quantity: '1'"), ...)

// ✅ Specific — asserts the positive case
assert.ok(booking.includes('quantity: String(totalPeople)'),
  'Service line item not using totalPeople')
Lesson: "Does not contain X" assertions are fragile — they pass when X is removed but also when X was never there, and they fail on unrelated uses of X. Prefer "contains Y" (the specific thing you want) over "does not contain X" (the thing you replaced).

TermDefinition in context
Discriminated unionA TypeScript type with a kind (or similar) tag field that narrows the type in a switch or if. The Flow type uses kind: 'public' | 'corporate' | ... so each branch gets narrowed state.
Immutable updateReplacing a value (especially an array or object) with a new copy rather than modifying it in place. Required by React's change-detection model.
Promise.allTakes an array of Promises and returns a single Promise that resolves when all of them resolve (or rejects as soon as any one rejects).
Graceful degradationA failure mode where the system does less but continues working. Here: if Square rate-limits customer creation, we fall back to creating only the primary customer instead of failing the whole booking.
catalogObjectIdA Square order line item field that points to an item in the Square Catalog. Square reads the catalog price for the charge — not a custom amount.

page.route()Playwright's API interception. Intercepts network requests matching a URL pattern and replaces the server response with mock data. Lets E2E tests run without a live backend.
strict mode violationPlaywright error thrown when a locator matches more than one element. The fix is always to make the selector more specific.
node:testNode.js built-in test runner (available since Node 18). No npm install needed. Used in verify-fixes.mjs for fast, dependency-free code-level assertions.
idempotency keyA unique ID sent with API calls to Square. If the same request is retried (due to a network error), Square detects the key and returns the original result instead of creating a duplicate record.
