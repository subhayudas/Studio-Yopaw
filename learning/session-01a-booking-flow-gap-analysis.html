Booking Flow:
Docs, Gap Analysis & Plan

📅 Session 01A 🔴 Intermediate ⏱ ~45 min read 📋 Part 1 of 2 🔗 Part B: Implementation

Session Overview

This session documents everything done during a major feature build for Studio Yopaw — a real puppy yoga studio in Saint-Lazare, QC, Canada. The studio uses a custom-built booking platform where customers select classes, enter contact information, accept a legal waiver, and pay via Square (a payment processor).

The session had two major phases (split into files A and B):

  • Part A (this file): We converted booking flow diagrams from images into written documentation, then ran a systematic gap analysis — comparing what the diagrams described the system should do against what the actual code actually does. We found 10 gaps.
  • Part B: We implemented all 10 fixes across 5 source files, then wrote and ran 67 automated tests (47 code-level + 20 browser-based) that all pass.
🔑Why This Matters

Gap analysis is a professional engineering discipline. Before writing any code, you identify the difference between design intent (what the feature is supposed to do) and implementation reality (what the code actually does). This prevents building on a broken foundation and is common practice at companies like Stripe, Shopify, and any team doing serious product work.

Learning Objectives

By the end of Part A you will be able to:

  • Convert a visual flow diagram into structured Markdown documentation
  • Perform a gap analysis between design and code
  • Read and understand a React Flow type using discriminated unions
  • Identify missing state, missing form fields, and missing API behaviour
  • Write an implementation plan with task ordering and dependencies

Recap Bridge — Where We Started

Studio Yopaw already had a working booking system. Customers could go to the website, pick a class type (Regular, Private, or Corporate), select a date and time, fill in their contact information, and either pay (Regular Class) or submit an inquiry (Private/Corporate). Square handles the payment processing; Resend sends email notifications; Zapier syncs leads to the studio's CRM.

The system was built iteratively, with features added one at a time. The visual design had evolved ahead of the code: someone had created detailed flow diagrams showing exactly how the booking process should work, but the code had been built from an earlier version of those designs. The diagrams lived only as images — there was no written documentation of the intended flow.

Our starting point: two images showing the intended booking flow and payment flow, and a codebase that partially matched those images but diverged in important ways nobody had catalogued.

Concept: Flow Documentation

Plain-English Explanation

A flow diagram is a picture showing every step a user takes through a feature — every decision point ("Do you have a mat?"), every branch ("Yes → go to date step / No → rent a mat"), and every terminal state ("Booking confirmed" or "Inquiry submitted").

Flow documentation converts that picture into text. Why? Because images are fragile — they get lost, they can't be searched, and developers can't paste them into code reviews. A text version lives in the repository, can be version-controlled, and can be linked from code comments.

Concept: Gap Analysis

Plain-English Explanation

A gap analysis is a structured comparison: "Here is what was designed. Here is what was built. What's different?"

Think of it like a house inspection. An inspector doesn't just say "the house looks fine" — they go room by room with a checklist, comparing the architectural drawings to what was actually constructed. They document every place the reality differs from the plan, assign severity (structural crack vs. missing light switch cover), and produce a report.

We did the same thing: we read the flow diagrams line by line, then opened the actual source code and checked whether each described feature existed.

How We Read the Code

The booking UI lives in src/App.tsx. The key thing to look for was the Flow type:

Concept: Discriminated Unions in TypeScript

Plain-English Explanation

A discriminated union (DU) is a TypeScript pattern for a variable that can hold different shapes of data. The "discriminator" is a field (always kind in this codebase) that tells you which shape you have.

Think of it like a shipping label. A box can contain a book, a lamp, or a bicycle. Each has different properties. The label on the outside says which kind of item is inside, and once you know the kind, you know exactly what properties exist.

TypeScript — analogy
// Without discriminated unions — messy, unclear
type Package = {
  kind: 'book' | 'lamp' | 'bicycle'
  isbn?: string      // only for books
  wattage?: number    // only for lamps
  wheelSize?: number  // only for bicycles
}

// With discriminated unions — TypeScript knows EXACTLY
// which fields exist based on the "kind" value
type Package =
  | { kind: 'book'; isbn: string }
  | { kind: 'lamp'; wattage: number }
  | { kind: 'bicycle'; wheelSize: number }

// Now TypeScript can catch errors like:
const pkg: Package = { kind: 'book', isbn: '978...' }
if (pkg.kind === 'book') {
  console.log(pkg.isbn)    // ✅ TypeScript knows this exists
  console.log(pkg.wattage) // ❌ TypeScript error — books don't have wattage
}

For the booking flow, this means that when flow.kind === 'public', TypeScript knows flow.step and flow.yoga exist. When flow.kind === 'chooseClass', those fields don't exist at all. This eliminates entire categories of bugs.

Concept: Multi-State React Components

The Problem with Complex State

The booking wizard has many moving parts that change independently: which step you're on, what date you selected, whether you need a mat rental, who the extra attendees are. Poor state management leads to bugs like "I went back a step but my date is still selected" or "The submit button is enabled when it shouldn't be."

🔑State Design Principle

Every piece of state should answer exactly one question. If you're storing "which step are we on" AND "what yoga style" in separate booleans, you can end up in impossible states (e.g., isYinStep=true and isGentleStep=true simultaneously). The discriminated union Flow type prevents this — only one shape can be active at a time.

The 10 Gaps We Found

After reading every part of the flow diagram and comparing it to the code line by line, we found these 10 gaps. We classified each by severity based on impact:

#GapWhereSeverity
1Extra attendees (up to 4) on Regular Class contact step — no UI, no stateFrontendHIGH
2Waiver checkbox per attendee — only one checkbox existed (primary booker only)FrontendHIGH
3Payment charged for 1 person only — should be price × attendee countFrontend + BackendHIGH
4Message / special request field missing from Private Class inquiryFrontend + BackendMEDIUM
5Company name field missing from Corporate inquiryFrontend + BackendMEDIUM
6Message / special request field missing from Corporate inquiryFrontend + BackendMEDIUM

10No booking note written to Square appointment (names + waiver status)BackendLOW

Why Some Gaps Are "High" Severity

Gaps 1–3 are HIGH because they affect revenue and user data. If a group of 3 friends books together:

  • Without gap 1 fixed: only 1 person can be registered — the others have no record
  • Without gap 2 fixed: only the primary booker signs the legal waiver — the others have no legal protection
  • Without gap 3 fixed: the studio only collects 1 person's fee instead of 3 — losing 67% of revenue for that booking

Why Did These Gaps Exist?

📖The Nature of Iterative Development

Almost every gap traces back to the same root cause: the booking flow was designed with multi-attendee support in mind, but the first implementation shipped with single-attendee support because "the simplest thing that works" is the correct starting point. Then the design evolved in the images/diagrams, but the code wasn't updated to match. This is completely normal — it's called design drift.

The lesson: diagrams and code are two different representations of the same system. Without regular reconciliation (exactly what we did in this session), they diverge. Good engineering teams schedule "design review" sessions specifically to catch this before it causes customer-facing bugs.

The Implementation Plan

We organized the 10 gaps into 6 tasks (A–F). Each task represents a coherent unit of work that can be reviewed independently. The tasks have dependencies — you can't test "payment multiplied by attendees" until you have the attendee count UI built.

TaskWhat It DoesDepends On
AAdd message + company name fields (Gaps 4, 5, 6)Nothing — isolated change
BAdd extra attendees UI (Gap 1)Task A (shares form patterns)
CPer-attendee waiver validation (Gap 2)Task B
DMultiply payment by attendee count (Gap 3)Task B
EPer-attendee Square customer creation + fallback (Gaps 7, 8)Task B + D
FAdd booking note with names + waiver status (Gaps 9, 10)Task B + E
💡Dependency Ordering

Task A and B can be started in parallel because they're independent. C, D, E, F must be sequential because each uses the extraAttendees state introduced in B. This is called a task dependency graph and is how engineering teams plan sprints in tools like Linear, Jira, or GitHub Projects.

Design Decisions

Max 4 extra attendees (5 total) Unlimited; max 20 Diagram specified 1–4 extras. Regular class seats are scarce — 5 spots per booking is reasonable. Groups larger than 5 must contact studio directly for a private event

Exercises

🧪🟢 Practice — Read the Flow Type

Open src/App.tsx and find the Flow type definition. Draw (on paper or in a tool) the complete state machine: all the valid transitions between states. For example: chooseClass → public.mat, public.mat → public.date, etc. Can you find any transitions that skip steps? Any back-navigation paths?

💡 Hint

Look for the goBack() function — it documents all reverse transitions. Look for every place setFlow() is called to find forward transitions.

🧪🟡 Stretch — Add a New Gap

The Private Event flow in the diagram shows "How many attendees?" but the design says attendees pay per-person too (same as Regular). Currently the Private Event inquiry doesn't go through Square at all — it's just an email to the studio. Write a gap description for this: what would be needed to make Private Events also charge online? List at least 5 things that would need to change.

💡 Hint

Think about: Square order creation, card form UI, waiver for all attendees, email confirmation vs. "we'll contact you" flow, whether the studio wants to handle the money now or later.

🧪🔴 Challenge — Design a Corporate Quote Flow

The current Corporate flow sends an inquiry with no price shown. Design (in writing — no code needed) a new flow that: (1) collects group size, (2) shows a price estimate based on group size × $46, (3) lets the user book and pay online. What changes to the Flow type would be needed? What new API endpoints? What Square API calls?

Glossary

TermDefinition
CRMCustomer Relationship Management — software that stores customer contact info and interaction history. Zapier syncs leads into Studio Yopaw's CRM.
Design DriftWhen the written/visual design of a feature diverges from its implementation over time, because both evolve independently without reconciliation.
Discriminated UnionA TypeScript type that can hold one of several specific data shapes, distinguished by a "kind" or "type" field. Prevents impossible states.
Gap AnalysisA structured comparison between intended design and actual implementation, producing a list of differences with severity ratings.
Idempotency KeyA unique string sent with API requests so that if the request is retried (due to network failure), the server knows to not process it twice. Used heavily with Square.
Rate LimitA restriction on how many API calls you can make in a time window. Square limits calls per second. If you exceed it, calls fail with HTTP 429.
ResendAn email delivery service (resend.com) used to send transactional emails (booking confirmations, inquiry notifications) programmatically.
Serverless FunctionA small piece of code that runs on demand in the cloud without a persistent server. Vercel hosts these as API endpoints in the api/ folder.
SquareA payment processing company. Studio Yopaw uses Square for payments, appointment booking, and customer management.
State MachineA design pattern where a system is always in one of a finite set of states. The Flow type implements a state machine for the booking wizard.
VercelA cloud hosting platform where both the React frontend and API functions are deployed. Handles HTTPS, CDN, and serverless execution.
ZapierA no-code automation platform. Studio Yopaw sends booking and inquiry data to Zapier webhooks, which forward them to a CRM or spreadsheet.
