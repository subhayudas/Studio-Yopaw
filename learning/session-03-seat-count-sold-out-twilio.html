A. What We Fixed & Built

But that booking note turned out to be more important than we realised. This session discovered and fixed three bugs, then replaced the entire email notification system with Twilio SMS (which works instantly on any phone with no email setup).

B. Recap Bridge

If you're reading this session fresh without the previous sessions, here's what was already built. Sessions 01a and 02 gave us a complete booking platform: a React frontend where customers choose a class type (Regular, Private, Corporate), pick a date and time, optionally add extra attendees (with per-person waivers), fill in contact details, and pay via Square's card form. The backend (Vercel serverless functions) creates a Square booking, charges the card, and sends notifications.

The critical detail from Session 02: when we create a Square booking for a group, we write a booking note onto the appointment: "Total attendees: 3 · Names: Alice Smith, Bob Jones, Carol Kim · Waiver confirmed: yes". This note is stored in Square's customerNote field and is readable back later via the bookings.list() API. We wrote it so the studio owner could see the group size in Square's calendar. But we're about to use it to fix our biggest bug.

One last piece of context: the availability API (GET /api/availability) works by listing all Square bookings for a date range, then for each scheduled slot computing seatsRemaining = maxSeats - bookedCount. It's this "bookedCount" computation that had the bug we're about to fix.

C. Bug 1 — Group Bookings Counted as 1 Seat

🐛 Bug Story #1: The Vanishing Seat Problem

🚨 The Symptom

A group of 5 people booked a Regular Class session. The /api/availability response showed that session as having 15 seats remaining (20 - 1 = 19... wait, no — 20 - 1 = 19 not 15). More specifically: a group of 5 people should have consumed 5 seats, leaving 15. But the API was reporting 19 seats remaining — as if only 1 person had booked.

This meant customers could overbook a session. If max capacity is 20 people but groups each counted as 1, you could fit 20 groups of 5 — that's 100 people in a room designed for 20.

🔍 The Diagnosis

In api/availability.ts, the booking count loop looked like this:

Before fix — api/availability.ts
for (const booking of bookings) {
  const st = booking.status ?? ''
  if (st === 'CANCELLED_BY_SELLER' || st === 'CANCELLED_BY_CUSTOMER' || st === 'DECLINED') continue
  const key = normalizeStartAt(booking.startAt)
  bookingCounts.set(key, (bookingCounts.get(key) ?? 0) + 1)
  //                                                              ^^^
  //  BUG: we always add 1, regardless of how many people are in this booking
}

Every Square booking record represents ONE appointment, regardless of how many people attend. Our code was counting booking records (1 per group), not counting actual people. A group of 10 was consuming the same "seat count" as a solo booking.

🧠 The Root Cause

Square's Booking object doesn't have an attendeeCount field — it just represents a calendar appointment. There is no built-in "number of people in this booking" in the Square API response. We had to find another way to know how many seats a booking consumed.

This is where Session 02's booking note saved us. Recall that in api/booking.ts we write:

api/booking.ts — booking note written in Session 02
const bookingNote = `Total attendees: ${totalPeople} · Names: ${allNames} · Waiver confirmed: yes`
// Example: "Total attendees: 3 · Names: Alice, Bob, Carol · Waiver confirmed: yes"

await square.bookings.create({
  booking: {
    customerNote: bookingNote,  // ← stored in Square, readable via bookings.list()
    // ... other fields
  }
})

The fix is to read this note back in availability.ts and parse the attendee count from it. This is a design pattern called metadata piggybacking — storing structured information in an unstructured text field because the API doesn't have a dedicated field for it.

🔧 The Fix

We added a small helper function that parses the booking note and returns the attendee count (defaulting to 1 if the note is absent or doesn't match the pattern):

api/availability.ts — new parseAttendeeCount helper
function parseAttendeeCount(note: string | undefined | null): number {
  if (!note) return 1          // no note → assume solo booking, deduct 1 seat
  const m = note.match(/Total attendees:\s*(\d+)/)
  // regex: find "Total attendees:" followed by optional whitespace and digits
  return m ? Math.max(1, parseInt(m[1], 10)) : 1
  // m[1] captures the number; Math.max(1, ...) prevents zero or negative counts
  // if no match → default 1 (backwards-compatible with old bookings)
}

Then the booking-counting loop was updated to add the attendee count instead of always adding 1:

api/availability.ts — booking count loop
const key = normalizeStartAt(booking.startAt) - bookingCounts.set(key, (bookingCounts.get(key) ?? 0) + 1) + bookingCounts.set(key, (bookingCounts.get(key) ?? 0) + parseAttendeeCount(booking.customerNote))

Additionally, the slot filtering logic changed so that all slots are returned — even ones with 0 seats — so the frontend can show a "Full" state:

api/availability.ts — always return the slot
const booked = bookingCounts.get(normalizeStartAt(startAt)) ?? 0 - const seatsRemaining = maxSeats - booked - if (seatsRemaining > 0) { - availabilities.push({ startAt, seatsRemaining }) - } + const seatsRemaining = Math.max(0, maxSeats - booked) + availabilities.push({ startAt, seatsRemaining }) // always include, even if 0

Why Math.max(0, ...)? Edge case protection: if somehow a booking was created that pushed booked over maxSeats (e.g. a race condition or a manually added Square appointment), we'd get a negative number without it. Negative seats remaining would be confusing — we clamp to 0.

🛡️ The Lesson

When you work with an API that doesn't have a field you need, look at what fields are available and consider storing metadata in them. The customerNote field is meant for "a note from the customer" but there's nothing stopping you from using it for structured data your own system writes — as long as you control both the writer and the reader.

The deeper lesson: write tests for domain invariants, not just code paths. We had tests that the booking flow worked, but no test that verified "if 5 people book, the seat count goes down by 5". Those are different things.

D. Bug 2 — Sold-Out Dates Vanished from the UI

🐛 Bug Story #2: The Disappearing Session

🚨 The Symptom

When a class was fully booked, its date row completely disappeared from the session picker. Customers had no way to know whether there was ever a session on that date, or whether they'd missed it. This caused confusion: "Is there a class on June 14th?" — impossible to tell from the UI.

🔍 The Diagnosis

In src/App.tsx, the effectiveDates computed value filtered the list of available dates. The old logic was:

Old effectiveDates filter (src/App.tsx)
const effectiveDates = useMemo(() => {
  return Object.keys(breedSchedule)
    .filter(date => {
      const entries = breedSchedule[date]
      if (!entries || entries.length === 0) return false
      return entries.some(e => e.serviceIds.length === 0 || e.serviceIds.includes(currentServiceVariationId))
      // ↑ only checked breed availability, NOT seat availability
      // a sold-out date with a matching breed STILL appeared in effectiveDates!
    })
    .sort()
}, [effectiveSlotsByDate, breedSchedule, currentServiceVariationId])
// The old filter used effectiveSlotsByDate in the dependency array
// but never actually used it in the filter logic — a disconnect.

Wait — where did the date rows go then, if effectiveDates included them? Because effectiveSlotsByDate itself only contained slots with seatsRemaining > 0 (before the availability.ts fix above). So when all slots were full, effectiveSlotsByDate[date] was an empty array, and the date row was rendered but immediately failed some other check.

Actually, after the availability.ts change (which now always returns slots, even 0-seat ones), effectiveDates needed to be updated to correctly distinguish "has available slots" from "all slots full". The two concepts had been mixed up.

🔧 The Fix

We split dates into two separate computed values:

src/App.tsx — effectiveDates (only truly available dates)
const effectiveDates = useMemo(() => {
  return Object.keys(effectiveSlotsByDate)
    .filter(date => {
      const entries = breedSchedule[date]
      if (!entries || entries.length === 0) return false
      const hasBreed = entries.some(e => e.serviceIds.length === 0 || e.serviceIds.includes(currentServiceVariationId))
      if (!hasBreed) return false
      return effectiveSlotsByDate[date].some(s => s.seatsRemaining > 0)
      // ↑ now we actually check: is there at least ONE slot with seats?
    })
    .sort()
}, [effectiveSlotsByDate, breedSchedule, currentServiceVariationId])

const soldOutDates = useMemo(() => {
  return Object.keys(slotsByDate)
    .filter(date => {
      const entries = breedSchedule[date]
      if (!entries || entries.length === 0) return false
      const hasBreed = entries.some(e => e.serviceIds.length === 0 || e.serviceIds.includes(currentServiceVariationId))
      if (!hasBreed) return false
      return slotsByDate[date].length > 0 && slotsByDate[date].every(s => s.seatsRemaining === 0)
      // ↑ ALL slots exist AND all are full → sold out
    })
    .sort()
}, [slotsByDate, breedSchedule, currentServiceVariationId])

Then the render loop was updated to combine both lists and display sold-out dates as non-interactive <div> elements (vs interactive <button> elements for available dates):

src/App.tsx — date list render (simplified)
{[...effectiveDates, ...soldOutDates].sort().map(dateIso => {
  const isSoldOut = soldOutDates.includes(dateIso)
  // sold-out dates render as a non-clickable div
  if (isSoldOut) {
    return (
      <div
        key={dateIso}
        className="pricing-session-row is-disabled"
        aria-disabled="true"
      >
        <span className="pricing-session-date">{formatShortSessionDate(dateIso, lang)}</span>
        <span className="pricing-session-spots">{s.pricingSpotFull}</span>
        {/* s.pricingSpotFull = "Full" in EN, "Complet" in FR */}
      </div>
    )
  }
  // available dates render as a clickable button
  return (
    <button key={dateIso} className="pricing-session-row" onClick={...}>
      ...
    </button>
  )
})}
button vs. div for interactivity. Using a <div> instead of a <button> for sold-out rows is intentional. A <button> is keyboard-focusable and screen-reader-announced as "clickable". A <div aria-disabled="true"> communicates "this exists but is not interactive". This is the correct ARIA (Accessible Rich Internet Applications) pattern for a disabled list item.

🛡️ The Lesson

"No data" and "data exists but is unavailable" are semantically different states and deserve different UI treatment. Hiding sold-out inventory is fine for a product catalogue where the user doesn't need to know something exists. But for a time-based calendar, customers benefit from seeing that a session happened/is scheduled — it helps them understand the studio's rhythm and plan ahead.

E. Bug 3 — Operator Precedence in Name Parsing

🐛 Bug Story #3: The Invisible Attendee Bug

🚨 The Symptom

When an extra attendee entered a single-word name (e.g. "Madonna" — no last name), the familyName field sent to Square was the first name repeated instead of the first name. So Square's customer record would show "Madonna Madonna" as first+last. Not a crash, but incorrect data in Square's CRM.

🔍 The Diagnosis

The name-splitting code:

The original (broken) line
return { givenName: parts[0] ?? a.name, familyName: parts.slice(1).join(' ') || parts[0] ?? a.name }
//                                                            ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^
//                                         This is: (A || B) ?? C  because || has HIGHER precedence than ??
//                                         intended: A || (B ?? C)

Let's trace through it with a single-word name "Madonna":

  • parts = ["Madonna"]
  • parts.slice(1).join(' ') → "" (empty string — no last name words)
  • "" || parts[0] ?? a.name is evaluated as ("" || parts[0]) ?? a.name
  • "" || "Madonna" → "Madonna" (correct so far)
  • "Madonna" ?? a.name → "Madonna" (fine)
  • Result: familyName: "Madonna" — first name used as last name too ✗

The intended logic was: "use slice(1).join(' ') as the last name IF it's non-empty, otherwise fall back to first name as a reasonable default." But the precedence made the fallback unreachable.

🔧 The Fix

Add parentheses to force the intended evaluation order:

api/booking.ts — extra attendee name parsing
- return { givenName: parts[0] ?? a.name, familyName: parts.slice(1).join(' ') || parts[0] ?? a.name } + return { givenName: parts[0] ?? a.name, familyName: parts.slice(1).join(' ') || (parts[0] ?? a.name) }

With parentheses: parts.slice(1).join(' ') || (parts[0] ?? a.name)

  • If slice gives a non-empty string (there IS a last name) → use it
  • If slice gives "" (no last name) → use parts[0] ?? a.name (i.e. the first word, or the whole name if splitting failed)

🛡️ The Lesson

JavaScript's ?? (nullish coalescing) and || (logical OR) have different precedences, and mixing them in a single expression without parentheses is a common footgun. The rule: || evaluates before ??. When you mix them, add parentheses to make your intent unambiguous — it's safer and more readable.

Never chain || and ?? in the same expression without explicit parentheses. ESLint rule no-mixed-operators catches this class of bug automatically — worth enabling.

J. Concept: Why Twilio SMS Instead of Email

The original implementation used Resend — a developer-friendly email API — to notify the studio owner of new bookings and inquiries. Here's why we switched to Twilio SMS:

DimensionEmail (Resend)SMS (Twilio)
Delivery latencySeconds to minutes~1–2 seconds
Open rate~20–30% within 1 hour~98% within 3 minutes
Inbox noiseCompetes with hundreds of emailsPhone buzzes, hard to miss
Setup neededEmail address + spam rulesPhone number (already have one)
Multiple recipientsCC/BCC or multiple sendsFan-out to N numbers trivially
Rich formattingHTML emailPlain text only
Cost~$0/month at low volume~$0.0075/SMS

For a small studio that books maybe 10–50 sessions per month, cost is negligible. The open rate and immediacy advantages far outweigh the loss of HTML formatting. Studio owners typically check phones constantly but might miss emails until they're at a desk.

Twilio works by giving you a virtual phone number that can send and receive SMS. You pay per message. Your backend calls the Twilio REST API (via the twilio npm package), which delivers the SMS through Twilio's carrier network.

Twilio terminology

  • Account SID — your unique account identifier (like a username). Starts with "AC".
  • Auth Token — the secret key for your account (like a password). Keep it secret.
  • From number — the Twilio virtual number that sends messages. Formatted in E.164 (e.g. +12494026223).
  • To number(s) — the real phone numbers that receive the messages. Also E.164.
  • E.164 — the international phone number format: + then country code then number, no spaces or dashes.

K. Building api/_twilio.ts

We created a shared module so all three API endpoints can import sendTeamSms() instead of each having their own Twilio client boilerplate. This is the DRY (Don't Repeat Yourself) principle applied to credential management.

api/_twilio.ts — full file
import twilio from 'twilio'           // official Twilio Node.js SDK
import { stripBom } from './_square.js'  // strips UTF-8 BOM from env vars on Windows

function getClient() {
  // Create a new Twilio client each call (stateless — safe in serverless)
  return twilio(
    stripBom(process.env.TWILIO_ACCOUNT_SID ?? ''),
    stripBom(process.env.TWILIO_AUTH_TOKEN ?? ''),
  )
}

function getTeamNumbers(): string[] {
  // TWILIO_TEAM_NUMBERS is comma-separated E.164 numbers
  // e.g. "+15142424947,+14385559999"
  return stripBom(process.env.TWILIO_TEAM_NUMBERS ?? '')
    .split(',')
    .map(n => n.trim())
    .filter(Boolean)          // remove empty strings from trailing commas
}

function getFromNumber(): string {
  return stripBom(process.env.TWILIO_FROM_NUMBER ?? '')
}

export async function sendTeamSms(body: string): Promise<void> {
  const client = getClient()
  const from = getFromNumber()
  const numbers = getTeamNumbers()

  if (!from || numbers.length === 0) {
    // Gracefully skip in dev if env vars not set — no crash
    console.warn('[Twilio] Skipping SMS — TWILIO_FROM_NUMBER or TWILIO_TEAM_NUMBERS not set')
    return
  }

  // Send to ALL team numbers in parallel — Promise.all waits for all to complete
  await Promise.all(
    numbers.map(to =>
      client.messages.create({ from, to, body }).catch(err =>
        console.error(`[Twilio] SMS to ${to} failed:`, err),
        // per-number catch: one failing number doesn't block others
      ),
    ),
  )
}
The _ prefix on _twilio.ts is a Vercel convention: files starting with _ in the api/ directory are treated as shared modules, not as HTTP endpoints. Without the prefix, Vercel would expose a /api/_twilio route — we definitely don't want that.

R. Design Decisions

DecisionAlternatives ConsideredWhy We Chose ThisTrade-offs
Read attendee count from booking note Store count in a custom Square field; count appointment segments We already write the note (Session 02), it's readable via the public API, and it's backwards-compatible with old single-person bookings Fragile if note format changes; but we control both writer and reader
Show sold-out dates as disabled rows Hide them entirely; show them with a "Join Waitlist" button Customers want to know sessions exist even if full — helps them plan. Waitlist adds complexity we don't need yet. Slightly more cluttered calendar; sold-out dates may confuse some users
Always return all slots from /api/availability (even 0 seats) Keep filtering at API level, add separate /api/sold-out endpoint Simpler: one endpoint, one source of truth. Frontend decides presentation. Slightly more data over the wire; frontend complexity increased slightly
Twilio SMS replacing Resend email Keep email; use both email + SMS; use WhatsApp Business SMS open rate ~98% vs email ~25%. Studio owner checks phone constantly. Single contact method is simpler. No rich HTML formatting; ~$0.0075 per SMS; need to manage Twilio number
sendTeamSms() fan-out (Promise.all) Single recipient; BCC-style chain Multiple team members need notifications. Promise.all sends in parallel — faster than sequential. If one number fails, Promise.all continues (per-number .catch ensures this)

W. Glossary

parseAttendeeCount
A helper function that reads the "Total attendees: N" string from a Square booking's customerNote field and returns N as a number, defaulting to 1.
E.164
International phone number format: +[country code][number], no spaces or special characters. Canadian: +15142424947.
Account SID
Twilio's unique identifier for your account, always starting with "AC". Like a username — safe to show in logs but not in client-side code.
Auth Token
Twilio's secret credential paired with Account SID. Never expose this in client-side code or logs.
Fan-out
Sending the same message to multiple recipients in parallel. Our sendTeamSms() fans out to all numbers in TWILIO_TEAM_NUMBERS using Promise.all.
is-disabled
A CSS class applied to sold-out date rows and full time slots to indicate they are not interactive. Combined with a div (not button) element and aria-disabled="true" for accessibility.
Operator Precedence
The order in which JavaScript evaluates operators in an expression. || evaluates before ?? — mixing them without parentheses can produce surprising results.
Nullish Coalescing (??)
JavaScript operator that returns the right-hand side only if the left is null or undefined. Unlike ||, it does NOT treat empty string ("") or 0 as falsy.
Logical OR (||)
JavaScript operator that returns the right-hand side if the left is falsy (null, undefined, "", 0, false). Has higher precedence than ??.
soldOutDates
A computed value in src/App.tsx containing dates where all time slots have seatsRemaining === 0. Displayed as disabled rows in the date picker.
effectiveDates
A computed value in src/App.tsx containing dates where at least one time slot has seatsRemaining > 0. Displayed as clickable rows in the date picker.
metadata piggybacking
Storing structured data in an unstructured text field (like a notes field) because the API doesn't have a dedicated field. Common workaround when APIs are limited. Must be carefully documented as a convention both the writer and reader follow.
DRY
Don't Repeat Yourself — a software principle. Instead of writing Twilio client code in booking.ts, inquiry.ts, AND square-webhook.ts, we extracted it into _twilio.ts so all three import it.
Vercel _prefix convention
Files in api/ starting with _ (underscore) are treated as shared modules, not HTTP endpoints. Without the prefix, Vercel would expose /api/_twilio as a public route.
ARIA
Accessible Rich Internet Applications — a set of HTML attributes (like aria-disabled="true") that communicate UI state to assistive technologies like screen readers.
