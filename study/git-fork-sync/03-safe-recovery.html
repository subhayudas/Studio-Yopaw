Step A — Backup before destructive commands

Commands like git reset --hard can permanently discard uncommitted changes. Even if you want to discard changes, a quick backup is often worth it (in case you change your mind).

Two common backup options
  • Backup branch: keeps your commits reachable (good for committed work)
  • File copy backup: saves working files to a folder (good for uncommitted work)

Step B — Abort an in-progress merge

If you’re mid-merge with conflicts and you want to “start over”, use: 

git merge --abort

This returns your working tree to the state it had before the merge started.

When git merge --abort is the right move
  • You started a merge accidentally
  • You want to discard the merge attempt and re-sync cleanly
  • You prefer to redo the merge later with a clearer plan

Step C — Align local branch to the fork branch

To “discard what I did locally” and match your fork exactly, you can hard-reset: 

git fetch origin
git reset --hard origin/main
Danger (but sometimes intended)

git reset --hard changes where your branch points and overwrites the working directory and index. Translation: your local uncommitted changes are gone.

In our session, we briefly created a backup branch first (so your local-only commit wasn’t lost), and later deleted it because you asked to discard changes.

A quick PowerShell note

In PowerShell, || isn’t a statement separator in many versions. Prefer: 

some-command
if ($LASTEXITCODE -ne 0) { "handle failure" }

That’s why we avoided Unix-style cmd || fallback.
