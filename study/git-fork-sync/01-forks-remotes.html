The model (keep this picture in your head)

A fork means you have two GitHub repos involved:

  • Upstream: the original/main repo you forked from (source of “official changes”)
  • Your fork: your copy on GitHub (where you push your work)

On your machine, Git stores those as remotes:

  • origin → usually your fork
  • upstream → usually the original repo
Why name them like this?

It’s a common convention. Git doesn’t require these names, but lots of tutorials assume them.

What we set in this repo

These are the remotes we configured: 

git remote -v
Important safety detail

We disabled pushes to upstream by setting its push URL to a dummy value: 

git remote set-url --push upstream DISABLE

Result: even if you accidentally run git push upstream main, it fails instead of pushing.

Remote branches vs local branches

When you run git fetch, Git downloads “snapshots” of branches from a remote and stores them locally as remote-tracking branches like origin/main and upstream/main.

Your working branch is typically a local branch like main.

Common confusion

main (local) and origin/main (remote-tracking) are not the same thing. They can point to different commits until you merge/rebase/reset.
