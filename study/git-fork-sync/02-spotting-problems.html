1) Check your current state

Start with: 

git status -sb

In our case earlier, it showed things like:

  • [ahead X, behind Y] → your local branch is not aligned with origin/main
  • UU some-file → unresolved merge conflicts (“unmerged” files)
What does ahead/behind mean?

Git compares commit history. “Ahead 1” means you have 1 local commit not in the remote branch. “Behind 18” means the remote has 18 commits you don’t have locally (yet).

2) Detect an in-progress merge/rebase

Sometimes Git is mid-operation (merge, rebase, cherry-pick). The easiest signal is:

  • git status says something like “You have unmerged paths” or “You are in the middle of a merge”

Another “low-level” way is checking these internal markers: 

git rev-parse -q --verify MERGE_HEAD
git rev-parse -q --verify REBASE_HEAD
git rev-parse -q --verify CHERRY_PICK_HEAD
git rev-parse -q --verify REVERT_HEAD

If a command prints a hash, it means that operation is in progress.

Why this matters

If you’re mid-merge, trying to “sync with upstream” can get messy because Git is already trying to combine two histories. First: return to a clean, consistent state.

3) Know your remotes

Always confirm which URL is origin and which is upstream: 

git remote -v
git remote show origin
git remote show upstream
Big risk

If you accidentally set origin to the upstream repo, git push could publish changes to the wrong place. That’s why we explicitly disabled pushes to upstream.
