The Problem: Ghost Appointments

The booking flow in api/booking.ts has three steps that must all succeed:

  1. Create the Square booking (adds the appointment to the calendar)
  2. Create a Square Order (computes taxes)
  3. Process payment (charges the card)

Before this fix, steps 2 and 3 ran without error handling. If payment was declined after the booking was already created in step 1, the appointment would sit in Square Appointments permanently with no associated payment — a ghost appointment that blocks the calendar slot for future customers.

The Fix: Cancel on Payment Failure

Steps 4 and 5 (Order + Payment) are now wrapped in a try/catch. If either throws, the booking created in step 3 is immediately cancelled:

// 3. Create booking first — this puts it in Square Appointments
const { booking } = await square.bookings.create({ ... })

let payment: ...
try {
  // 4. Create Order (with taxes)
  if (gstTaxId && qstTaxId) {
    const { order } = await square.orders.create({ ... })
    chargeAmount = order!.totalMoney!.amount!
    orderId = order!.id
  } else {
    chargeAmount = BigInt(baseAmountCents)
  }

  // 5. Process payment
  const result = await square.payments.create({ ... })
  payment = result.payment

  if (payment!.status !== 'COMPLETED' && payment!.status !== 'APPROVED') {
    throw new Error(`Payment not approved (status: ${payment!.status})`)
  }
} catch (paymentErr) {
  // Payment failed — cancel the booking to avoid ghost appointment
  console.error('Payment failed — cancelling booking', {
    bookingId: booking!.id,
    error: paymentErr instanceof Error ? paymentErr.message : paymentErr,
  })
  try {
    await square.bookings.cancel({
      bookingId: booking!.id!,
      idempotencyKey: randomUUID(),
      bookingVersion: booking!.version,
    })
  } catch (cancelErr) {
    // Both payment AND cancellation failed — manual cleanup needed
    console.error('CRITICAL: Payment failed AND booking cancellation failed', {
      bookingId: booking!.id,
      cancelErr,
    })
  }
  throw paymentErr  // re-throw so the outer catch returns an error response
}
The CRITICAL log means a ghost appointment exists in Square Appointments that could block future bookings for that time slot. If you see this in Vercel logs, go to Square Dashboard → Appointments and manually cancel/delete the appointment with that bookingId.

Why Create the Booking Before the Payment?

This is counterintuitive — why not process payment first, then create the booking? The comment in the code explains:

"Create the booking first so we have an ID to link the order and payment against. This is what makes Square's calendar display the booking as paid rather than 'balance due'."

The Order's referenceId and the Payment's referenceId both point to booking.id. Square uses these references to correlate the payment with the appointment in its Appointments dashboard. If the order/payment were created without this reference, Square would show the appointment as unpaid even if the card was charged successfully.

Payment Status Validation

After calling square.payments.create(), the handler now explicitly checks the returned status:

if (payment!.status !== 'COMPLETED' && payment!.status !== 'APPROVED') {
  throw new Error(`Payment not approved (status: ${payment!.status})`)
}

This catches cases where Square returns a 200 response but with a status like PENDING or FAILED. Without this check, a technically-failed payment could be treated as a success.

Friendly Payment Error Messages

Before, the error response was:

return res.status(500).json({ error: err.message })

This would expose raw Square error strings like "Status code: 402 Body: {\"errors\":[{\"code\":\"GENERIC_DECLINE\",...}]}" directly to the user.

Now, the error goes through friendlyPaymentError(err):

function friendlyPaymentError(err: unknown): string {
  if (!(err instanceof Error)) return 'Booking failed. Please try again.'

  // Parse Square SDK error body: "Status code: 4xx Body: {...}"
  const bodyMatch = err.message.match(/Body:\s*(\{[\s\S]*\})\s*$/)
  if (bodyMatch) {
    try {
      const parsed = JSON.parse(bodyMatch[1])
      const code = parsed.errors?.[0]?.code ?? ''
      const category = parsed.errors?.[0]?.category ?? ''

      if (code === 'GENERIC_DECLINE' || code === 'CARD_DECLINED')
        return 'Your card was declined. Please check your details or try a different card.'
      if (code === 'INSUFFICIENT_FUNDS')
        return 'Insufficient funds. Please try a different card.'
      if (code === 'CARD_EXPIRED')
        return 'Your card has expired. Please use a different card.'
      if (code === 'CVV_FAILURE')
        return 'The security code (CVV) is incorrect. Please try again.'
      if (code === 'ADDRESS_VERIFICATION_FAILURE')
        return 'Address verification failed. Please check your billing address.'
      if (code === 'INVALID_CARD')
        return 'The card number is invalid. Please check and try again.'
      if (category === 'PAYMENT_METHOD_ERROR')
        return 'Payment declined. Please check your card details and try again.'
      if (category === 'RATE_LIMIT_ERROR')
        return 'Too many attempts. Please wait a moment and try again.'
    } catch {
      // JSON parse failed — fall through to generic message
    }
  }

  return 'Booking failed. Please try again.'
}

Booking Status & Timestamp Normalization

Both api/booking.ts (race-condition seat check) and api/availability.ts (listing booked slots) had two related bugs fixed:

1. Wrong cancellation status strings

In the seat-check inside booking.ts:

- b.status !== 'CANCELLED'
+ b.status !== 'CANCELLED_BY_SELLER' &&
+ b.status !== 'CANCELLED_BY_CUSTOMER'

The string 'CANCELLED' is not a valid Square booking status. The real ones are CANCELLED_BY_SELLER and CANCELLED_BY_CUSTOMER. Without this fix, cancelled bookings would still count toward the seat limit, and a cancelled class slot would appear full.

In availability.ts the same logic was applied when counting booked seats, plus DECLINED was also excluded.

2. Timestamp fractional seconds mismatch

// Normalize before comparing:
const norm = (s: string) => s.replace(/\.\d+Z$/, 'Z')
const taken = bookings.filter(
  b => norm(b.startAt ?? '') === norm(startAt) && ...
).length

Square sometimes returns timestamps with fractional seconds (2026-06-14T14:30:00.000Z) and sometimes without (2026-06-14T14:30:00Z). A direct string comparison would miss matches. The norm() helper strips the fractional part before comparing.
