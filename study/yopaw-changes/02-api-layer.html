The Big Picture: From Square Catalog to a JSON File

Before this batch of changes, the session schedule was stored in Square Catalog as a special item named "Class Session Dates" under a "class schedule" category, with one catalog variation per date. The availability API fetched this catalog entry to know which dates were valid, and the breeds API fetched all-day bookings in Square to find which dog breed was attending each session (using customer given/family name as the EN/FR breed name).

This was clever but fragile — it depended on obscure Square Catalog conventions and a custom naming convention for customers. Both APIs were rewritten to read a plain class-schedule.json file that lives in the repo root.

Result: Zero extra Square API calls for schedule/breed data. The schedule can be updated by editing one JSON file and deploying. Breed information is explicit and bilingual.

To update the schedule: edit class-schedule.json, commit, and deploy. Changes are live immediately — no Square Dashboard work needed.

vercel.json — includeFiles

Vercel serverless functions run from a bundle. Static files not referenced by import/require are not included by default. class-schedule.json is loaded with readFileSync (a runtime FS read), so it must be explicitly listed:

"functions": {
"api/**/*.ts": {
"memory": 256,
- "maxDuration": 15
+ "maxDuration": 15,
+ "includeFiles": "class-schedule.json"
}
}

Without this, readFileSync(...'class-schedule.json'...) would throw ENOENT in production.

api/availability.ts — Full Rewrite

Before

  1. Called square.catalog.list({types:['CATEGORY']}) to find the "class schedule" category.
  2. Called square.catalog.list({types:['ITEM']}) to find the "Class Session Dates" item under that category.
  3. Extracted allowed dates from catalog item variation names.
  4. Called square.bookings.searchAvailability() with serviceVariationId and optional teamMemberId to get Square's own availability view.
  5. Listed existing bookings to subtract counts from Square's slots.
  6. Merged Square slots with synthetic slots for any catalog-date/time combos Square didn't know about.

Required query params: serviceVariationId, startDate, endDate, teamMemberId.

After

  1. Reads class-schedule.json synchronously with readFileSync.
  2. Filters schedule.dates to the requested range.
  3. For each date × time pair, converts the Montreal-local time to a UTC ISO string (DST-aware, same algorithm as before).
  4. Lists existing Square bookings to subtract booked seats. This step is now non-fatal — if the Square API is unavailable, all seats appear available (better UX than a hard error).
  5. Returns only slots with seatsRemaining > 0.

Required query params: startDate, endDate only. serviceVariationId is no longer needed.

Booking status fix: the old code checked booking.status === 'ACCEPTED'. The new code skips CANCELLED_BY_SELLER, CANCELLED_BY_CUSTOMER, and DECLINED, which are the actual Square cancellation enum values. Also normalises timestamps to strip fractional seconds before comparing (/\.\d+Z$/ → 'Z').

Montreal → UTC Time Conversion (DST-aware)

// Given a date like "2026-06-14" and a time like "10:30" (Montreal local):
const [h, m] = timeStr.split(':').map(Number)
const utcGuess = new Date(`${dateIso}T${h}:${m}:00Z`)          // pretend it's UTC
const checkTime = slotMontrealTime(utcGuess.toISOString())  // what Montreal time is this?
const [ch, cm] = checkTime.split(':').map(Number)
const offsetMinutes = (h * 60 + m) - (ch * 60 + cm)        // compute the offset
const corrected = new Date(utcGuess.getTime() + offsetMinutes * 60_000)
const startAt = corrected.toISOString().replace(/\.\d{3}Z$/, 'Z') // strip ms

This handles both EDT (UTC−4) and EST (UTC−5) correctly because slotMontrealTime uses Intl.DateTimeFormat with timeZone: 'America/Toronto'.

Cache Header

A new Cache-Control: no-store header was added to the response. This prevents stale availability data from being cached by the browser or CDN between requests.

Better Error Logging

The catch block now logs structured error details instead of just the raw error object:

console.error('availability error', {
  message: (err as Error)?.message,
  statusCode: (err as { statusCode?: number })?.statusCode,
  errors: (err as { errors?: unknown })?.errors,
})

api/breeds.ts — Full Rewrite

Before (Square all-day bookings approach)

  1. Split the date range into ≤30-day chunks.
  2. Called square.bookings.list() for each chunk, filtered to allDay === true && status === 'ACCEPTED' bookings.
  3. Fetched each such booking's customerId from square.customers.get().
  4. Used customer.givenName as the EN breed name and customer.familyName as the FR breed name.
  5. Extracted serviceVariationId from appointment segments to filter by service type.

Required making N+1 Square API calls (one per unique customer ID). Was async.

After (class-schedule.json breeds field)

export default function handler(req: VercelRequest, res: VercelResponse) {
  // ...
  const schedule = loadSchedule()
  const breeds = schedule.breeds ?? {}
  const result: Record<string, { breed: {en: string; fr: string}; serviceIds: string[] }[]> = {}

  for (const [date, breed] of Object.entries(breeds)) {
    if (date < startDate || date > endDate) continue
    if (!breed.en && !breed.fr) continue
    result[date] = [{ breed, serviceIds: [] }]
  }
  return res.status(200).json({ schedule: result })
}

The handler is now synchronous (async keyword removed). Zero Square API calls. Response is instant.

serviceIds: [] means "all services". In the front-end rendering logic, the breed is shown for a date if e.serviceIds.length === 0 || e.serviceIds.includes(currentServiceVariationId). An empty array matches any service — so the breed info shows up regardless of which yoga style the user is booking.

Front-End Breed Filter Fix (src/App.tsx)

Two places in App.tsx filter breed entries by service ID. Both were updated to accept serviceIds: [] as "matches all":

In the availableDates memo

- return entries.some(e => e.serviceIds.includes(currentServiceVariationId))
+ return entries.some(e => e.serviceIds.length === 0 || e.serviceIds.includes(currentServiceVariationId))

In the date row rendering

- .filter(e => e.serviceIds.includes(currentServiceVariationId))
+ .filter(e => e.serviceIds.length === 0 || e.serviceIds.includes(currentServiceVariationId))

Without this fix, the breed badge would never appear because the new API always returns serviceIds: [].

vite.config.ts — Dev API Plugin

Previously, testing the booking flow locally required running vercel dev (which spins up a local Vercel runtime). Now, vite dev alone is sufficient for UI development.

A custom Vite plugin devApiPlugin() was added that registers Connect middleware for /api/breeds and /api/availability:

Structure

function devApiPlugin(): Plugin {
  return {
    name: 'dev-api',
    configureServer(server) {
      server.middlewares.use('/api/breeds', (req, res) => {
        // reads class-schedule.json, filters by date range, returns schedule
      })
      server.middlewares.use('/api/availability', (req, res) => {
        // reads class-schedule.json, generates synthetic UTC slots
      })
    },
  }
}

export default defineConfig({
  plugins: [react(), tailwindcss(), devApiPlugin()],
})
Dev-only. This plugin is part of the Vite config, so it only runs during vite dev. Production uses the real Vercel functions in api/. The dev plugin intentionally uses Intl.DateTimeFormat directly (rather than slotMontrealTime from _config.js) because that helper is a CJS module not available in the Vite context.

Montreal → UTC in the Dev Plugin

function montrealLocalToUtc(dateIso: string, timeStr: string): string {
  const [h, m] = timeStr.split(':').map(Number)
  const utcGuess = new Date(`${dateIso}T${h}:${m}:00Z`)
  const parts = new Intl.DateTimeFormat('en-CA', {
    timeZone: 'America/Toronto',
    hour: '2-digit', minute: '2-digit', hour12: false,
  }).formatToParts(utcGuess)
  const localH = Number(parts.find(p => p.type === 'hour')?.value ?? h)
  const localM = Number(parts.find(p => p.type === 'minute')?.value ?? m)
  const offsetMinutes = (h * 60 + m) - (localH * 60 + localM)
  const corrected = new Date(utcGuess.getTime() + offsetMinutes * 60_000)
  return corrected.toISOString().replace(/\.\d{3}Z$/, 'Z')
}
