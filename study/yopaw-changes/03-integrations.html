Zapier Webhooks

Webhook URLs (defined as constants)

ConstantURL (partial)When it fires
ZAPIER_NEW_CONTACT_URL …/4oigr6o/ New Square customer created in booking.ts, or any inquiry submitted (inquiry.ts).
ZAPIER_NEW_BOOKING_URL …/4oig0ml/ After a successful Square payment in booking.ts.
ZAPIER_FORM_URL …/4ok9t5x/ Contact form submitted (both confirmed booking and inquiry paths).

These constants are defined in both api/booking.ts and api/inquiry.ts (duplicated intentionally — the two files are independent Vercel functions).

Why fire early (before payment)?

The most important change is that Zapier is called when the contact form is submitted, not just when payment completes. If a customer fills in their name/email/phone and then abandons the payment step, the lead is still captured. This is the "Move Zapier fetch before waiver gate" commit (1c2b47e).

Zapier in api/booking.ts

Three webhooks fire at different points during the booking server flow:

1. After new Square customer is created

// Only fires for brand-new customers (not returning ones)
await fetch(ZAPIER_NEW_CONTACT_URL, {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({
    firstName: givenName,
    lastName: familyName,
    fullName: `${givenName} ${familyName}`,
    email,
    phone,
    source: 'booking',
  }),
}).catch((err) => console.error('[Zapier] new-contact webhook failed:', err))

2. After successful payment — booking confirmation

await fetch(ZAPIER_NEW_BOOKING_URL, {
  method: 'POST',
  body: JSON.stringify({
    firstName: givenName,
    lastName: familyName,
    fullName: `${givenName} ${familyName}`,
    email,
    phone,
    bookingId: booking!.id,
    startAt,
    serviceName,
    totalCents: Number(chargeAmount),
    totalDollars: (Number(chargeAmount) / 100).toFixed(2),
    paymentStatus: payment!.status,
  }),
}).catch(...)

3. After successful payment — form-style fields

await fetch(ZAPIER_FORM_URL, {
  method: 'POST',
  body: JSON.stringify({
    firstName: givenName,
    lastName: familyName,
    phone,
    email,
    dateWanted: startAt,
    eventType: serviceName,
    peopleCount: groupSize ?? '1',
  }),
}).catch(...)

All Zapier calls use .catch(() => {}) / .catch(err => console.error(...)) — they are fire-and-forget. A Zapier failure never breaks the booking response.

Zapier in api/inquiry.ts

The inquiry endpoint (Private Event and Corporate) also fires Zapier after sending the Resend notification email:

const [firstName, ...rest] = fullName.trim().split(/\s+/)
const lastName = rest.join(' ')

// Form fields
await fetch(ZAPIER_FORM_URL, {
  method: 'POST',
  body: JSON.stringify({
    firstName, lastName, phone, email,
    dateWanted: preferredDate ?? '',
    eventType: classType,
    peopleCount: groupSize ?? '',
  }),
}).catch(...)

// New contact
await fetch(ZAPIER_NEW_CONTACT_URL, {
  method: 'POST',
  body: JSON.stringify({
    firstName, lastName, fullName,
    email, phone, classType,
    preferredDate: preferredDate ?? '',
    preferredTime: preferredTime ?? '',
    groupSize: groupSize ?? '',
    source: 'inquiry',
  }),
}).catch(...)

Note: ZAPIER_NEW_BOOKING_URL is not called from inquiry — that webhook is reserved for confirmed paid bookings only.

Zapier in src/App.tsx (Frontend Lead Capture)

The most important Zapier change is that the frontend also fires the inquiry API as soon as the contact form is submitted — before the user even sees the payment step.

In submitPublic() (yin & gentle flows)

const submitPublic = (e: FormEvent) => {
  e.preventDefault()
  if (flow.kind !== 'public') return

  // ← Fires FIRST, before waiver gate
  fetch('/api/inquiry', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({
      fullName,
      email,
      phone,
      classType: flow.yoga === 'gentle' ? 'Private Event' : 'Regular Class',
      preferredDate: selectedSessionIso ?? '',
      preferredTime: selectedTimeSlotId ?? '',
      groupSize: flow.yoga === 'gentle' ? privateGroupCount : '',
    }),
  }).catch(() => {})

  const needsWaiver = flow.yoga !== 'gentle'
  if (needsWaiver && !waiverAccepted) return  // ← waiver gate is here

  // ... advance to payment
}
Even if the waiver hasn't been accepted yet (and the function returns early), the fetch('/api/inquiry') has already been dispatched. So clicking "Submit" captures the lead regardless of whether they complete the booking.

In submitCorporate()

const submitCorporate = (e: FormEvent) => {
  e.preventDefault()

  fetch('/api/inquiry', {
    method: 'POST',
    body: JSON.stringify({
      fullName: corpName,
      email: corpEmail,
      phone: corpPhone,
      classType: 'Corporate',
      preferredDate: selectedSessionIso ?? '',
      preferredTime: selectedTimeSlotId ?? '',
      groupSize: privateGroupCount,
    }),
  }).catch(() => {})

  // ... advance to payment step (which is the confirmation screen for corporate)
}

Google Ads Conversion Tracking

Global site tag in index.html

<!-- Google Ads conversion tracking -->
<script async src="https://www.googletagmanager.com/gtag/js?id=AW-18168099243"></script>
<script>
  window.dataLayer = window.dataLayer || [];
  function gtag(){dataLayer.push(arguments);}
  gtag('js', new Date());
  gtag('config', 'AW-18168099243');
</script>

The account ID is AW-18168099243. This loads the Google Ads global site tag on every page view.

TypeScript declaration in src/App.tsx

declare function gtag(...args: unknown[]): void

This tells TypeScript that gtag is a global function (injected by the script tag), preventing a type error when it's called.

Conversion event fired on contact form submit

// Inside submitPublic(), after the waiver check passes:
if (typeof gtag !== 'undefined') {
  gtag('event', 'conversion', { send_to: 'AW-18168099243/YPflCLS3wLAcEKvjnNdD' })
}

The conversion label is YPflCLS3wLAcEKvjnNdD. The typeof gtag !== 'undefined' guard prevents errors if the script tag fails to load (e.g., blocked by an ad blocker).

Timing: The conversion fires after the waiver check (when the user advances to the payment step), not when the form is first submitted. This means it fires for users who actually intend to pay, not for people who abandon at the waiver.
