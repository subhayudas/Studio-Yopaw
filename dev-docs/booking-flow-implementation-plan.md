## Detailed Gap Analysis

### Gap 1 — Extra attendees not supported (Regular Class)

**Diagram says:** Contact step includes "Add extra attendees (1–4)" — additional people can join the primary booker.

**Current code (`App.tsx` line 1078–1197):** The contact form for `yin` only collects `fullName`, `email`, `phone` for a single person. There is no UI for adding attendees. `extraAttendees` state doesn't exist.

**Impact:** A user can only book for themselves. The diagram explicitly allows 1 primary + up to 4 extras (5 total).

---

### Gap 2 — Single waiver checkbox, not one per attendee

**Diagram says:** "Checkbox per attendee. All must be checked to proceed." The payment flow image header also confirms: "Each card has its own waiver checkbox."

**Current code (`App.tsx` line 1141–1176):** Only one `waiverAccepted` boolean exists. One checkbox for the primary booker.

**Impact:** If extra attendees are added (Gap 1), each would need their own waiver. Until Gap 1 is implemented this blocks the full intended UX. Even for a single person the current single checkbox is structurally fine, but won't scale.

---

### Gap 3 — Payment not multiplied by attendee count

**Diagram says:** Payment = "Price × number of attendees — charged via Square."

**Current code (`App.tsx` line 1205–1209):**
```ts
const groupSize = flow.yoga === 'gentle' ? Math.max(2, parseInt(privateGroupCount || '2', 10)) : 1
const baseCents = svc.baseAmountCents * groupSize + matRentalCents
```
For `yin` (Regular Class), `groupSize` is hardcoded to `1`. The payment summary and the amount sent to Square always represent a single person.

**Current `booking.ts` line 85–92:** The order is created with `quantity: '1'` for the service line item, and `attendeeCount: 1` is sent to Zapier (line 185). No attendee count is passed from frontend at all.

**Impact:** If 3 people attend, only 1 person's price is charged. This is a revenue and accuracy bug once Gap 1 is implemented.

---

### Gap 4 — No "Message / special request" field on Private Class inquiry

**Diagram says:** Private Class inquiry form includes "Message / special request."

**Current code (`App.tsx` line 1078–1197, `submitPublic` line 686–716):** The `gentle` path contact form only has `fullName`, `email`, `phone`. No message field exists. The `POST /api/inquiry` call (line 692–714) also doesn't send a `message` field. `inquiry.ts` doesn't accept or display one.

---

### Gap 5 — No "Company name" field on Corporate inquiry

**Diagram says:** Corporate inquiry form includes "Company name."

**Current code (`App.tsx` line 1286–1343):** Corporate contact form collects `corpName`, `corpEmail`, `corpPhone` — no company/organization name. `submitCorporate` (line 744–769) sends only `fullName`, `email`, `phone` to `/api/inquiry`. `inquiry.ts` doesn't accept or display a `companyName` field.

---

### Gap 6 — No "Message / special requests" field on Corporate inquiry

**Diagram says:** Corporate inquiry form includes "Message / special requests."

**Same situation as Gap 5** — no message field in the form or API.

---

### Gap 7 — `booking.ts` creates only one Square customer

**Diagram says:** "Try: create client per attendee — Square Customers API — one call each."

**Current `booking.ts` lines 44–58:** The code does one `customers.search()` by email, creates one customer if not found, and uses that single `customerId` throughout. There is no loop over attendees.

**Impact:** Extra attendees (once Gap 1 is added) will not be registered as individual Square customers.

---

### Gap 8 — No API-limit fallback path

**Diagram says:** "If API limits: fallback — Create primary client only."

**Current `booking.ts`:** No try/catch or retry logic around the customer creation loop. If Square rate-limits, the whole booking fails instead of falling back gracefully.

---

### Gap 9 — Booking appointment has no attendee count

**Diagram says:** "Book appointment — attendee count = total people."

**Current `booking.ts` line 79–93:** `square.bookings.create()` sends `appointmentSegments` with one entry but the booking object has no field that records total attendee count. Square's Appointments API supports a `customerNote` or a `note` field on the booking which could carry this, but nothing is sent.

---

### Gap 10 — No booking note on appointment

**Diagram says:** "Booking note on appointment (always): Total attendees: N · Names · Waiver confirmed: yes"

**Current `booking.ts`:** The `booking` object sent to `square.bookings.create()` has no `customerNote`, `sellerNote`, or equivalent. The studio has no way to see attendee details directly in their Square calendar.

### Task B — Add extra attendees to Regular Class contact step (Gap 1)

**Effort:** Medium. New UI section in the contact step; new state to track up to 4 extra people.

#### B1 — State (`src/App.tsx`)

```ts

const [extraAttendees, setExtraAttendees] = useState<ExtraAttendee[]>([])
```

Reset `extraAttendees` in `chooseClass()` and `restartPublicBooking()`.

#### B2 — UI — "Add extra attendees" section in contact form

Below the phone field (yin path only), add:

```tsx
{/* Extra attendees — up to 4 */}
<div className="pricing-extra-attendees">
  <p className="pricing-helper-text">
    {s.pricingExtraAttendeesHelper} {/* "Add up to 4 more people" */}
  </p>
  {extraAttendees.map((a, i) => (
    <div key={i} className="pricing-extra-attendee-card">
      <input
        type="text"
        placeholder={s.pricingExtraAttendeeName} /* "Attendee name" */
        value={a.name}
        onChange={e => {
          const next = [...extraAttendees]
          next[i] = { ...next[i], name: e.target.value }
          setExtraAttendees(next)
        }}
        required
      />
      <label className="pricing-checkbox-row">
        <input
          type="checkbox"
          checked={a.waiverAccepted}
          onChange={e => {
            const next = [...extraAttendees]
            next[i] = { ...next[i], waiverAccepted: e.target.checked }
            setExtraAttendees(next)
          }}
        />
        <span>{s.pricingWaiverConsentPrefix}…</span>
      </label>
      <button type="button" onClick={() => setExtraAttendees(extraAttendees.filter((_, j) => j !== i))}>
        {s.pricingRemoveAttendee} {/* "Remove" */}
      </button>
    </div>
  ))}
  {extraAttendees.length < 4 && (
    <button
      type="button"
      onClick={() => setExtraAttendees([...extraAttendees, { name: '', waiverAccepted: false }])}
    >
      {s.pricingAddAttendee} {/* "+ Add attendee" */}
    </button>
  )}
</div>
```

### Task D — Multiply payment by total attendees (Gap 3)

Depends on Task B (need the attendee count).

#### D1 — Frontend payment summary (`src/App.tsx` ~line 1204)

```ts
const totalPeople = 1 + extraAttendees.length   // primary + extras
const baseCents = svc.baseAmountCents * totalPeople + matRentalCents
```

Update the summary label to show `${totalPeople} × Puppy Yoga Class`.

#### D2 — Backend (`api/booking.ts`)

1. Destructure `extraAttendees` from `req.body`.
2. Compute `totalPeople = 1 + (extraAttendees?.length ?? 0)`.
3. Change the service line item quantity from `'1'` to `String(totalPeople)`:
   ```ts
   { quantity: String(totalPeople), catalogObjectId: serviceVariationId, … }
   ```
   Square will compute the correct total automatically — taxes will scale with it.
4. Update Zapier payload: `attendeeCount: totalPeople`.
5. Update the notification email to show all attendee names.

> Note: `baseAmountCents` sent from the frontend is display-only and is not used for the charge amount — the charge always comes from `order.totalMoney.amount`. The quantity field is what drives the actual total.

### Task E — Create Square customer per attendee + fallback (Gaps 7, 8)

Depends on Task B (need attendee list).

#### E1 — Backend (`api/booking.ts`) — replace single customer create

Current pattern (single customer):
```ts
let customerId = searchResult.customers?.[0]?.id
if (!customerId) { await square.customers.create(…) }
```

New pattern (one per attendee with fallback):
```ts
async function findOrCreateCustomer(given: string, family: string, email: string, phone: string) {
  const search = await square.customers.search({
    query: { filter: { emailAddress: { exact: email } } },
  })
  const existing = search.customers?.[0]?.id
  if (existing) return existing
  const { customer } = await square.customers.create({
    idempotencyKey: randomUUID(),
    givenName: given, familyName: family, emailAddress: email, phoneNumber: phone,
  })
  return customer!.id!
}

// Build attendee list: primary + extras
const allAttendees = [
  { givenName, familyName, email, phone },
  ...(extraAttendees ?? []).map((a: { name: string }) => {
    const parts = a.name.trim().split(' ')
    return { givenName: parts[0] ?? a.name, familyName: parts.slice(1).join(' ') || parts[0] ?? a.name, email, phone }
  }),
]

// Try to create all customers; fallback to primary only on rate-limit
let customerIds: string[]
try {
  customerIds = await Promise.all(
    allAttendees.map(a => findOrCreateCustomer(a.givenName, a.familyName, a.email, a.phone))
  )
} catch (err) {
  console.warn('Customer batch creation hit API limit — falling back to primary client only', err)
  const primarySearch = await square.customers.search({
    query: { filter: { emailAddress: { exact: email } } },
  })
  const primaryId = primarySearch.customers?.[0]?.id
    ?? (await square.customers.create({
        idempotencyKey: randomUUID(),
        givenName, familyName, emailAddress: email, phoneNumber: phone,
      })).customer!.id!
  customerIds = [primaryId]
}

const customerId = customerIds[0]  // primary customer used for booking/payment
```
