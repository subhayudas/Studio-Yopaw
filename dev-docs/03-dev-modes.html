The Three Dev Modes

Why npm run dev, vercel dev, and production are three completely different environments — and when you need each one.

The three modes at a glance

Mode 1 — Fast Dev

Vite Dev Server

npm run dev

Only the React frontend runs. API endpoints are either mocked or unavailable. Hot reload active. Fastest to start.

Mode 2 — Full Dev

Vercel Dev

vercel dev

Both the React frontend AND the serverless functions run locally. Slower to start. Needed for testing real payments, emails, webhooks.

Mode 3 — Live

Production on Vercel

git push → auto-deploy

Built React app served as static files. Functions run on Vercel's actual serverless infrastructure. Real money, real emails.

Mode 1: npm run dev — Vite dev server

When you type npm run dev, you are starting only Vite. Vite spins up a local web server (usually on port 5173) that serves your React app with instant hot reload.

$ npm run dev

  VITE v8.x  ready in 312 ms

  ➜  Local:   http://localhost:5173/
  ➜  Network: http://192.168.x.x:5173/

What is actually running when you open localhost:5173:

  • Vite serves your index.html and TypeScript/React files directly (no build needed — Vite transforms files on the fly)
  • When you change a .tsx file and save, Vite sends the update to your browser over a WebSocket — the browser hot-swaps the component in under 100ms
  • Requests to /api/*? They go to the Vite server, which has NO idea what to do with them — unless you add custom middleware (which is exactly what this project does — see page 4)
Python analogy npm run dev is like running python -m http.server 5173 — but smarter, because it compiles TypeScript on the fly and does hot reload. It is NOT running your API routes.

Mode 2: vercel dev — the full local simulator

vercel dev is a separate CLI tool from Vercel that simulates the entire Vercel platform on your machine. It:

  • Starts the Vite frontend (same as npm run dev)
  • Also starts a local Node.js server that handles /api/* requests by running your api/*.ts functions
  • Reads your vercel.json config and applies the same routing rules as production
  • Injects environment variables from your .env.local file
$ vercel dev

Vercel CLI 40.x
> Running Dev Command "vite"
> Ready! Available at http://localhost:3000

Now /api/booking actually works — it runs your TypeScript code
with real Square API calls, real Resend emails, etc.

The downside: vercel dev is slower to start because it is compiling your TypeScript API functions and running them inside a Node.js process. For every API request, it re-executes the function (similar to the production serverless behavior).

Why not always use vercel dev? Because vercel dev simulates real API calls. If you're testing the UI flow for a booking, you don't want to accidentally charge a real card or create real Square bookings. Using npm run dev with mocked API responses is safer for UI development.

Mode 3: Production — deployed on Vercel

When you push to GitHub (or run vercel --prod), this is what happens on Vercel's servers:

What Vercel does automatically on every git push:

1. npm run build
   └─ tsc -b (TypeScript type check)
   └─ vite build (compiles React → dist/)

2. Bundle each api/*.ts file
   └─ Each file becomes a separate Lambda function
   └─ class-schedule.json is bundled in (per vercel.json)

3. Deploy:
   └─ dist/ → Vercel's global CDN (static files)
   └─ api/* → Vercel's serverless function runtime

A CDN (Content Delivery Network) is a network of servers distributed worldwide. Your dist/ files (HTML, CSS, JS) get copied to servers in 40+ cities. When a user in Paris opens the site, they download the files from a server in Frankfurt, not from wherever your Vercel project is hosted. This is why static sites load so fast globally.

Why does having two dev modes make sense?

This pattern — a fast frontend-only server AND a full-stack simulator — is very common in modern web development. Here is the reasoning:

90% of development time is spent on UI work: making a button look right, fixing a layout, adjusting the booking flow steps. For all of that, you don't need real API calls. You just need the UI to render with some fake data.

10% of development time requires real API testing: verifying that the Square payment actually goes through, checking that the Resend email arrives, testing the webhook signature validation. For that, you use vercel dev.

Keeping the fast dev server for the 90% case makes daily development much more pleasant.

How Python projects handle the same problem A Python FastAPI app might use pytest with a test client that mocks external API calls (using unittest.mock or respx). The concept is the same: you want most of your development loop to be fast and isolated from external services.

What the npm scripts actually run

Open package.json and you'll see a "scripts" section. These are just shortcuts:

{
  "scripts": {
    "dev": "vite",
    // ↑ "npm run dev" just runs "vite"

    "build": "tsc -b && vite build",
    // ↑ "npm run build" runs TypeScript compiler THEN Vite build

    "preview": "vite preview",
    // ↑ Serves the built dist/ folder locally (not dev server)
    //   Useful to test the production build before deploying

    "lint": "eslint ."
    // ↑ Checks code style with ESLint (like flake8 or ruff in Python)
  }
}

npm (Node Package Manager) is the JavaScript equivalent of pip. package.json is the equivalent of pyproject.toml or requirements.txt — it lists dependencies and scripts.

Request flow in each mode

Mode 1: npm run dev (frontend request)

Browser ──GET /api/availability──▶ Vite dev server :5173 ──▶ devApiPlugin() middleware ──reads──▶ class-schedule.json
/api/booking would return 404 — no handler for it in Vite

Mode 2: vercel dev (full request)

Browser ──POST /api/booking──▶ vercel dev router :3000 ──▶ api/booking.ts (Node.js) ──▶ Square API (real/sandbox)
vercel dev intercepts the request and routes to the correct function

Mode 3: Production

Browser ──GET /──▶ Vercel CDN (global) ──▶ dist/index.html (static)
Browser ──POST /api/booking──▶ Vercel Serverless ──spawns──▶ api/booking.ts (Lambda)
