The Clever Trick in This Project

How devApiPlugin() in vite.config.ts fakes two backend endpoints so you can develop 90% of the UI without ever needing vercel dev.

The problem this solves

When you run npm run dev, Vite's dev server does not know anything about your api/ folder. If the React app makes a request to /api/availability, the Vite server would return a 404 error.

You have two options:

Option A: Always use vercel dev

Slow to start (3–10 seconds). Real API calls hit Square/Resend on every test. Risk of accidentally creating real bookings or sending real emails during UI development.

Option B: Teach Vite to fake the API

Fast dev server stays fast. No external API calls during UI work. Safe to run repeatedly. This is what this project does — the devApiPlugin() in vite.config.ts.

What is "middleware"?

A web server receives HTTP requests. Middleware is a function that sits between the incoming request and the final handler — it can inspect the request, modify it, respond to it directly, or pass it along.

Think of it like a security guard at a building entrance. Every visitor goes through the guard before reaching their destination. The guard can let them through, turn them away, or handle them directly.

Python analogy (WSGI middleware) In Python's Flask/FastAPI, middleware is the @app.before_request hook or a WSGI middleware wrapper. It intercepts every request before your route handlers run. In Express.js (Node.js), you write app.use('/path', handler). Vite's dev server uses the same concept — it is built on Connect (a minimal Node.js middleware framework), so you can plug in your own handlers.

The devApiPlugin() — what it does

In vite.config.ts, there is a function called devApiPlugin() that returns a Vite plugin object. A Vite plugin is just an object with specific properties that Vite recognises and hooks into its lifecycle.

This particular plugin uses the configureServer hook, which lets you add custom middleware to the Vite dev server. The plugin registers two middleware handlers:

  • One that intercepts GET /api/breeds
  • One that intercepts GET /api/availability

When the React app makes either of those requests, Vite intercepts them before any 404 happens, reads class-schedule.json from disk, computes the response, and returns it directly — no network call to any external server, no Node.js serverless function, just local JSON processing.

Code walkthrough — annotated

vite.config.ts — the plugin function
function devApiPlugin(): Plugin {
  return {
    name: 'dev-api',
    // "configureServer" is a Vite lifecycle hook that runs when the
    // dev server starts. "server" is the Vite dev server instance.
    configureServer(server) {

      // ── BREEDS ENDPOINT ────────────────────────────────────────────
      server.middlewares.use('/api/breeds', (req, res) => {
        // This function runs EVERY TIME someone requests /api/breeds
        // during development. It reads the JSON file from disk,
        // filters by date range, and returns the result.

        const schedule = loadSchedule()
        const url = new URL(req.url, 'http://localhost')
        const startDate = url.searchParams.get('startDate')
        const endDate = url.searchParams.get('endDate')

        // Build the response object (same shape as the real API)
        const result = {}
        for (const [date, breed] of Object.entries(schedule.breeds)) {
          if (date < startDate || date > endDate) continue
          result[date] = [{ breed, serviceIds: [] }]
        }

        res.setHeader('Content-Type', 'application/json')
        res.end(JSON.stringify({ schedule: result }))
      })

      // ── AVAILABILITY ENDPOINT ──────────────────────────────────────
      server.middlewares.use('/api/availability', (req, res) => {
        // Generates time slots from the schedule file.
        // In production, this ALSO checks Square for existing bookings.
        // Here in dev, we just return all seats as available (maxSeats).

        const schedule = loadSchedule()
        const availabilities = []

        for (const date of schedule.dates) {
          for (const time of schedule.times) {
            availabilities.push({
              startAt: montrealLocalToUtc(date, time),
              seatsRemaining: schedule.maxSeats   // always full in dev
            })
          }
        }

        res.setHeader('Content-Type', 'application/json')
        res.end(JSON.stringify({ availabilities }))
      })
    }
  }
}
The dev mock is slightly different from production In production, /api/availability checks Square's API for existing bookings and subtracts them from maxSeats. The dev mock always returns all seats as available because there is no Square API call. This is intentional — during UI development you want to always be able to see and click on available dates.

What is a "Connect middleware"?

Vite's dev server uses a Node.js library called Connect under the hood. Connect is a very minimal HTTP server framework. server.middlewares.use(path, handler) is Connect's way of saying "when a request comes in for this path, run this handler function."

The handler function receives two arguments: req (the incoming request — URL, headers, body) and res (the response object — you write your reply into this). This is the same pattern used by Express.js and many other Node.js frameworks.

Why only mock two endpoints?

Look at the full list of API endpoints — only /api/breeds and /api/availability are mocked in the Vite plugin. Why not the others?

✓

/api/availability — mocked because it's read-only UI data

Shows which dates have open seats. Needed to render the calendar. Has no side effects. Safe and useful to mock.

✓

/api/breeds — mocked because it's read-only UI data

Shows which dog breed appears on each session date. Needed for the date picker badges. Has no side effects. Safe and useful to mock.

✗

/api/booking — NOT mocked (has real side effects)

Charges a real credit card. Creates a real Square booking. Sends a real email. You only want to test this with vercel dev against Square's sandbox environment.

✗

/api/inquiry — NOT mocked (sends real emails)

Sends a real email via Resend. Fires a Zapier webhook. You only test this with vercel dev.

✗

/api/square-webhook — NOT mocked (receives external calls)

This endpoint receives calls FROM Square's servers. There's nothing to mock locally — Square has to be able to reach your server, which requires a public URL (or a tunnel tool like ngrok).

The data flow from file to screen

When you open the booking section on localhost:5173 and the calendar of available dates appears, here is the exact chain of events:

1

React renders the booking UI

The PricingSection component mounts. It calls useSquareAvailability(), a custom React hook.

2

The hook fetches the API

useSquareAvailability calls fetch('/api/availability?startDate=...&endDate=...') — this is a normal browser HTTP request.

3

Vite's middleware intercepts it

The request hits the Vite dev server. The devApiPlugin middleware matches /api/availability and runs the handler function — no external network call needed.

4

JSON is read from disk and returned

class-schedule.json is read, slots are computed, and a JSON response is sent back to the browser — all in milliseconds.

5

React updates the UI

The hook receives the slot data, updates state, and React re-renders the calendar showing the available dates.

For your portfolio site — do you need any of this?

Probably not much of it. Here is what applies and what doesn't:

What you'll likely use

Vite as your build tool. npm run dev for development. npm run build to generate your static site. Deploying the dist/ folder to Vercel. Maybe one simple serverless function for a contact form.

What you probably won't need

vercel dev (unless you add real API functionality). The devApiPlugin() trick (that's for when you have complex API mocking needs). The entire api/ folder structure (unless adding backend features later).

The simplest Vercel portfolio deployment 1. Build your site with npm run build (or just write plain HTML/CSS/JS — no build needed)
2. Push to GitHub
3. Connect the repo on vercel.com — click Deploy
4. Done. Free HTTPS, global CDN, auto-redeploys on every push.

You do NOT need vercel.json, api/ folders, serverless functions, or any of the complexity in this project to deploy a portfolio site on Vercel.
