How a traditional backend server works

If you built a backend in Python, you would probably write something like this with Flask or FastAPI:

# A traditional Flask server
from flask import Flask, request, jsonify

app = Flask(__name__)

@app.route('/api/availability')
def get_availability():
    return jsonify({ 'slots': [...] })

@app.route('/api/booking', methods=['POST'])
def create_booking():
    data = request.get_json()
    return jsonify({ 'bookingId': '123' })

# This server runs CONTINUOUSLY, 24/7,
# waiting for requests to come in.
app.run(port=5000)

The server starts up, loads everything into memory, and stays running forever. Even when nobody is using it at 3am, your server process is alive, using memory, consuming CPU cycles (even if just idle), and costs money every hour.

What is "serverless"?

Serverless is a confusing name — there absolutely is a server. What it means is: you don't manage a continuously-running server process. Instead, your code runs only when a request comes in, and stops immediately after.

Lifecycle of a serverless function call

User hits /api/booking — request arrives at Vercel's infrastructure

Vercel spins up a container — a tiny, isolated execution environment starts from scratch in <1 second

Your function runs — Node.js loads api/booking.ts, runs the handler, returns a response

Container shuts down — everything is discarded. No memory persists between calls.

You are billed only for execution time — typically milliseconds. Vercel's free tier is extremely generous.

Python analogy for serverless Imagine instead of a running Flask server, each of your route functions is a separate script. When a request comes in, the system runs python booking.py, waits for it to finish, and returns the output. When it's done, the process exits. That is essentially what serverless is.

How Vercel Functions work

Vercel looks at your api/ folder and automatically converts each .ts file into a serverless function. You do not configure routes manually — the filename IS the route.

api/
  ├── availability.ts → becomes GET /api/availability
  ├── booking.ts → becomes POST /api/booking
  ├── inquiry.ts → becomes POST /api/inquiry
  ├── breeds.ts → becomes GET /api/breeds
  ├── square-webhook.ts → becomes POST /api/square-webhook
  ├── _square.ts → NOT a route (underscore = private helper)
  └── _config.ts → NOT a route (underscore = private helper)

Files starting with _ (underscore) are private helpers — Vercel does not create endpoints for them. They are just TypeScript modules that the other functions import.

What a function file looks like

Each function file exports one default function — the handler. Compare this to a Flask route handler:

Python (Flask route)
from flask import request, jsonify

# This is ONE route in a larger app
@app.route('/api/availability')
def availability():
    start = request.args.get('startDate')
    end = request.args.get('endDate')
    slots = compute_slots(start, end)
    return jsonify({ 'availabilities': slots })
TypeScript (Vercel Function)
// This IS the entire file: api/availability.ts
import type { VercelRequest, VercelResponse }
    from '@vercel/node'

export default async function handler(
    req: VercelRequest,
    res: VercelResponse
) {
    const { startDate, endDate } = req.query
    const slots = await computeSlots(startDate, endDate)
    res.json({ availabilities: slots })
}

Both do the same thing: receive an HTTP request, do some work, return a JSON response. The structure is nearly identical — just different syntax.

Cold starts — the main trade-off of serverless

The first time a function is called after it has been idle, Vercel needs to spin up a new container. This takes extra time — usually 200ms to 1 second. This is called a cold start. Subsequent requests within a short window hit a warm container and respond much faster.

For a booking site with low traffic, this is usually acceptable. For a high-traffic API needing sub-100ms response times on every request, you might need a traditional always-running server instead.

No persistent memory between requests Because each invocation may get a fresh container, you cannot store data in a global variable and expect it to persist. Every call to api/booking.ts starts with a clean slate. This is why the app uses Square's API (external persistent storage) for bookings, not an in-memory dictionary.

Environment Variables — secrets on the server

Your functions need API keys (Square, Resend, etc.) but you cannot put these in your source code — anyone could see them on GitHub. The solution is environment variables: values set directly in Vercel's dashboard that get injected into your function at runtime.

Python (os.environ)
import os

api_key = os.environ.get('SQUARE_API_KEY')
# Set in shell: export SQUARE_API_KEY=abc123
# Set in .env file with python-dotenv
TypeScript (process.env)
// Identical concept, different syntax:
const apiKey = process.env.SQUARE_API_KEY

// Set in Vercel dashboard (production)
// Set in .env.local file (local dev)

There are two types of env vars in this project:

  • Server-only (no prefix) — only available in api/*.ts functions. Never sent to the browser. Example: SQUARE_ACCESS_TOKEN.
  • Browser-safe (VITE_ prefix) — baked into the compiled JavaScript that Vite sends to the browser. Anyone can read them in DevTools. Only put public IDs here, never secrets. Example: VITE_SQUARE_APP_ID.

vercel.json — the deployment configuration

This small config file controls how Vercel deploys the project. Let's read through this project's vercel.json:

{
  // Apply these settings to every file inside api/
  "functions": {
    "api/**/*.ts": {
      "memory": 256,         // Max RAM per function: 256 MB
      "maxDuration": 15,     // Kill the function if it runs > 15 seconds
      "includeFiles": "class-schedule.json"
      // ↑ Bundle this JSON file INTO the function package
      // (otherwise functions can't read it at runtime)
    }
  },

  // URL rewrite rules:
  "rewrites": [
    // /api/* → route to the matching serverless function
    { "source": "/api/(.*)", "destination": "/api/$1" },

    // Everything else (/, /about, /refund-policy, etc.)
    // → serve index.html (SPA routing trick)
    { "source": "/((?!src/|node_modules/|...).*)",
      "destination": "/index.html" }
  ]
}

The rewrite rule that sends everything to index.html is what makes the SPA routing work. When someone types www.yopaw.ca/refund-policy in their browser, Vercel serves index.html — then React's JavaScript reads the URL and shows the correct page component. Without this rule, a direct URL visit would return a 404 because there is no actual file at /refund-policy.

All API endpoints in this project

File Method URL What it does
availability.ts GET /api/availability Returns available time slots (reads schedule, checks Square for existing bookings)
breeds.ts GET /api/breeds Returns dog breed for each session date (reads schedule JSON)
booking.ts POST /api/booking Creates a Square booking, charges the card, sends confirmation email
inquiry.ts POST /api/inquiry Saves a lead inquiry, sends email via Resend, fires Zapier webhook
square-webhook.ts POST /api/square-webhook Receives events FROM Square (payment completed, customer created)
For your portfolio site You might only need one serverless function — something like api/contact.ts that receives a contact form submission and sends you an email. Everything else (your bio, projects, etc.) can be pure static HTML. This is the perfect use case for Vercel's free tier.
