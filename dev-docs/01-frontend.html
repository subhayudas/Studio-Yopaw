The Frontend — What runs in the browser

React, TypeScript, Vite, and the "build" process — explained for a Python developer who has never touched a JavaScript framework.

What is "the frontend"?

The frontend is everything that runs inside the user's web browser. It is HTML (structure), CSS (styling), and JavaScript (behaviour). The browser downloads these files from a server and runs them locally on the user's machine — your server sends the files once, and the browser does the rest.

The backend is code that runs on a server. The user never sees it. It handles things like saving data to a database, charging a credit card, or sending an email.

Studio Yopaw is a Single-Page Application (SPA). That means the entire website is one HTML file (index.html). When you navigate to a "new page" (like going from the home section to the pricing section), JavaScript is swapping out the content on the screen — no new file is fetched from the server. This is why it feels so fast.

Python analogy A SPA is like a Python program that keeps running and redraws its terminal UI — vs a CGI script that starts fresh on every request. The browser downloads the program once and runs it continuously.

TypeScript — JavaScript with type hints

JavaScript has no type system. You can write let x = 5 and then later do x = "hello" and JavaScript won't complain until it crashes at runtime. TypeScript is a superset of JavaScript (meaning all valid JavaScript is also valid TypeScript) that adds a type system.

Python (type hints)
def greet(name: str) -> str:
    return f"Hello, {name}"

# mypy catches this at check time:
greet(42)  # error: int, not str
TypeScript (enforced types)
function greet(name: string): string {
    return `Hello, ${name}`
}

// TypeScript catches this at compile time:
greet(42)  // error: number, not string

The key difference: TypeScript code cannot run directly in a browser. It must be compiled (converted) to plain JavaScript first. Vite does this compilation step for you automatically.

For your portfolio site You don't have to use TypeScript. Plain JavaScript works fine. TypeScript is a quality-of-life tool for larger projects where you want your editor to catch mistakes as you type.

React — what it actually does

React is a JavaScript library that lets you write your UI as a tree of components. A component is just a JavaScript function that returns some HTML-like syntax. Here is the actual relationship:

Python (Jinja2 template)
# A reusable "component"
def render_button(label, color):
    return f"""
    <button style="color:{color}">
        {label}
    </button>
    """
React (JSX component)
// A reusable component
function Button({ label, color }) {
    return (
        <button style={{ color }}>
            {label}
        </button>
    )
}

The HTML-looking syntax inside JavaScript is called JSX (JavaScript XML). It is not real HTML — it compiles to JavaScript function calls. Vite's React plugin handles this transformation.

React's big idea: instead of manually manipulating the HTML on screen (like jQuery used to do), you describe what the screen should look like based on your data, and React figures out the minimal changes needed to make the real HTML match. This is called the Virtual DOM.

Vite — the build tool and dev server

Vite (pronounced "veet", French for "fast") is two things in one:

  1. A development server — when you run npm run dev, Vite starts a local web server. You open http://localhost:5173 in your browser and see the site. This is only for development; it is not the same as how the site runs in production.
  2. A bundler / build tool — when you run npm run build, Vite compiles all your TypeScript, transforms JSX, processes Tailwind CSS, and combines everything into a tiny set of plain HTML/CSS/JS files. These go into the dist/ folder and are what Vercel actually deploys.
Python analogy for Vite's dev server Running npm run dev is similar to running uvicorn main:app --reload or flask run — a local server that serves your app and restarts when you change files. But Vite is much faster because it doesn't re-bundle everything on every change — it only updates the parts that changed.
Studio-Yopaw/
  ├── vite.config.ts <-- Vite's configuration file (like setup.py or pyproject.toml)
  ├── src/ <-- Your source TypeScript/React files
  │   ├── App.tsx <-- Root React component
  │   └── main.tsx <-- Entry point (like if __name__ == "__main__")
  ├── index.html <-- The single HTML file (the shell)
  └── dist/ <-- Output after "npm run build" (what gets deployed)

What "the build" actually means

When you type npm run build, this is what happens step by step:

# This is what npm run build does under the hood:
tsc -b        # TypeScript compiler checks for type errors
vite build    # Vite bundles everything into dist/

The dist/ folder that comes out looks something like:

dist/
  ├── index.html <-- The entry HTML, now with script/link tags injected
  └── assets/
      ├── index-[hash].js <-- All your React code, compiled + minified
      └── index-[hash].css <-- All your Tailwind CSS, compiled + purged

The [hash] in filenames is a fingerprint of the file's contents. If the file changes, the hash changes, which forces browsers to download the new version instead of using their old cached copy. This is called cache busting.

Important: the browser never sees your .tsx files TypeScript (.tsx) files only exist during development. The browser only ever downloads the compiled JavaScript from dist/assets/. If you open DevTools in the browser and look at Sources, you'll see index-[hash].js, not App.tsx.

Hot reload / HMR (Hot Module Replacement)

When you save a file during npm run dev, your browser updates instantly without a full page refresh. This is called HMR (Hot Module Replacement). Vite sends only the changed module to the browser over a WebSocket (a persistent connection), and React swaps it in without losing the current UI state.

This is one of Vite's killer features — it makes the development loop extremely fast compared to older tools that had to re-bundle the entire app on every save.

Summary — Python vs JavaScript frontend

Python web stack
# Development:
flask run
# or
uvicorn main:app --reload

# Template rendering on server:
return render_template('page.html')

# No "build step" — Python runs
# your source directly
JavaScript frontend stack
# Development:
npm run dev  # starts vite dev server

# Rendering happens in browser:
# React runs JS → updates DOM

# Build step required for production:
npm run build  # compiles .tsx → .js

The fundamental difference: Python renders HTML on the server and sends finished HTML to the browser. React renders HTML in the browser using JavaScript. Both approaches are valid — they just have different trade-offs.
